## 对话 2：质疑「复杂性」

**KawOwl → Claude**
> 为什么说两套逻辑共存协调起来会很复杂？如果已经有预置文本，就跳过一些环节，这样的想法有哪些问题？

**Claude → KawOwl**

分析了三个具体摩擦点：

1. **驱动模型差异**：AI 模式是自动循环（每 2s 一轮），预置模式应该是玩家点击推进——但这好解决
2. **记忆连续性**：预置文本跳过 AI 但仍需 ingest 到记忆——也好解决
3. **过渡判定**：什么时候从预置切到 AI、什么时候切回——这是真正的设计工作量，但它是一个局部模块

**结论：承认之前说的「很复杂」说得过重了。核心改动是 runOneCycle() 顶部加一个分支，真正的工作量在过渡管理器和锚点收敛逻辑上。**

## 对话 4：要求深入传统引擎数据结构

**Claude → KawOwl**（提问）
> 预置场景的数据格式用哪种方案？方案 A: presetScenes（SceneOutput[]）还是 方案 B: narrative + dialoguePresets？
> AI 生成的场景也要改成点击推进吗？

**KawOwl → Claude**
> 等一下，我还想深入讨论一下传统引擎的数据结构，可能影响决策。

**Claude → KawOwl**

深入分析了四个引擎的内部数据模型：

- **Ren'Py**：编译成 AST，每个语句是 Node（SayNode / SceneNode / MenuNode），核心是有序 Node 列表，用 `who` 字段区分旁白和对话
- **ink**：编译为嵌套 JSON 容器，文本是 `^前缀` 字符串，指令是操作码，线性执行
- **Monogatari**：JSON 数组中每项是指令，字符串 = 对话/旁白，对象 = 选择/条件
- **Tuesday.js**：三层嵌套 block → scene → dialog[]

**提炼出共同范式**：所有引擎都是「有序指令序列」——内容和指令混在同一个列表里，角色对话在叙事流中有精确位置，最小播放单元是一条指令。

**关键结论**：修正方案.md 将 narrative 和 dialoguePresets 分离存储有根本问题——旁白和对话在叙事中交织，拆开就失去节奏。SceneOutput[] 恰好就是这个「有序指令列表」。

## 对话 5：记忆归属——引出参与框架

**KawOwl → Claude**
> 你分析得对，我想问的正是传统引擎的数据结构中是不是还隐含了哪句台词是谁说的这个信息？因为我想如果要做到记忆的话，后续角色是要想起来自己说过什么话的。

**Claude → KawOwl**

指出传统引擎都在数据中标记了说话者（Ren'Py 的 SayNode.who、Monogatari 的角色前缀、Tuesday.js 的 character 字段）。但传统引擎没有记忆系统——它们是无状态回放机。

我们的引擎不同：当前 core-loop 的记忆入库把所有东西压成纯文本字符串，丢失了「谁说的」「谁听到的」结构化信息。SceneOutput 的 speaker 字段可以利用起来做更细粒度的记忆归类。

## 对话 6：四维元数据

**KawOwl → Claude**
> 我再重复一下，你看我说得对不对。关于一句话，要记录下来是谁说了，说给谁听，实际上谁听到了，在什么场合说的。

**Claude → KawOwl**

确认了这四个维度，并引入了 Goffman (1981) 的参与框架（Participation Framework）理论来命名这四个维度：

| 维度 | 术语 | 含义 |
|------|------|------|
| 谁说了 | speaker | 发话者 |
| 说给谁听 | addressee | 受话者 |
| 实际上谁听到了 | witnesses / overhearers / eavesdroppers | 旁听者层级 |
| 在什么场合 | setting | 时间+地点+事件 |

用血色多瑙河的巷道截获场景做了具体映射示例。

## 对话 7：展开参与框架理论

**KawOwl → Claude**
> 你说的「用语言学的术语来说，这就是一句话的参与框架（Participation Framework）」，很有意思，能不能展开说说？

**Claude → KawOwl**

展开了 Goffman 的完整理论：

**说话者侧**拆为三层：Animator（发声者）→ Author（措辞者）→ Principal（立场者）。大多数时候三者合一，但莎乐美引用弗洛伊德时她是 Animator，弗洛伊德是 Principal。

**听话者侧**拆为层级树：
```
所有在场的人 (participants)
├── 被认可的参与者 (ratified participants)
│   ├── 受话者 (addressed recipient)
│   └── 未被直接点名的参与者 (unaddressed recipient)
└── 旁观者 (bystanders)
    ├── 有意的旁听者 (overhearers) — 说话者知道他们在
    └── 偷听者 (eavesdroppers) — 说话者不知道他们在
```

指出这对谍战题材特别有意义——同一句话在不同角色的记忆里应该不同（speaker / addressee / overhearer / eavesdropper 各自视角）。

提出了数据结构方向：SceneOutput 增加 `addressee` + `witnesses` 字段，引擎运行时根据角色位置计算 overhearers / eavesdroppers。

## 对话 8：参与框架 vs visibleTo

**KawOwl → Claude**
> 如何用参与框架来描述 visibleTo？我感觉参与框架这个模型的描述能力会比 visibleTo 强一点。

**Claude → KawOwl**

分析了 `visibleTo` 和参与框架的关系：

- `visibleTo` 是**事件级别**的标记（哪条视角线能看到这个事件）
- 参与框架是**逐句级别**的（每句话的感知角色可以不同）
- `visibleTo` 是参与框架的**投影**：`event.visibleTo = 去重(所有 PF 中出现的角色)`

在修正方案.md 明确要求的「严格单一 POV」前提下，元叙事可见性 = 故事层可感知性，因此 **visibleTo 可以由 PF 完全推导，不需要单独存储**。

提出了两层默认 + 覆盖的简化模型：事件级默认用 `affectedCharacters`，逐句覆盖只在需要精确控制时标注 `addressee` 和 `witnesses`。

## 对话 9：记忆投影的精确模型

**KawOwl → Claude**
> 记忆入库时，可能不止这么简单。每个人的记忆入库，应该都是叙事框架经过它自己的可见性过滤后的投影。以上面的举例……假设受话者 A 只是不知道有人偷听，那么除了 eavesdroppers 的信息，它都应该加入记忆。

**Claude → KawOwl**

确认了精确模型：**每个角色的记忆 = 完整场景经过该角色的可见性滤镜后的投影**。这个滤镜不只过滤内容，还过滤「谁在场」这个信息本身。

具体规则：

| 角色身份 | 内容 | 对「谁在场」的认知 |
|---------|------|-----------------|
| Speaker / Addressee / Unaddressed | 完整 | ratified ∪ overhearers（不含 eavesdroppers）|
| Overhearer | 可能残缺 | ratified ∪ overhearers（不含 eavesdroppers）|
| Eavesdropper | 可能残缺 | ratified ∪ overhearers（不含其他 eavesdroppers）|

关键洞察：阿尔伯特的记忆里记着「只有我和莎乐美知道这件事」——但实际上戈林在偷听。这个认知差异就是信息不对称的核心。

扩展了 EpisodicMemory 结构，增加 `pfId`（引用回源 PF）、`participationRole`、`knownPeers`、`involvedCharacters`。

## 对话 10：PF 为信源，记忆为派生视图

**KawOwl → Claude**
> 数据结构可能还要再思考下。比如说我们共用一个「参与框架」的持久化数据。所有相关人存储「参与框架」id + filterContent 结果 + filterPeers 结果。我的主要思路是想让数据在运行时能引用原始信源，并且做可再现的计算。在剧本文件中，这个数据是全面的「参与框架」，再加上一些用来计算的元数据。

**Claude → KawOwl**

确认了正规化设计：

```
剧本文件（单一事实源）
  └── ParticipationFrame
        ├── 完整内容
        ├── 参与者清单
        └── 计算元数据

运行时记忆（派生视图）
  └── EpisodicMemory
        ├── pfId → 引用回源 PF
        ├── filteredContent → 缓存（可重算）
        └── knownPeers → 缓存（可重算）
```

核心性质：**给定 pfId + characterId，任何时候都能重新计算。filteredContent 是缓存，不是信源。**

好处：可溯源、可验证、一致性（改原文所有投影自动更新）、省空间。

提出了 ParticipationFrame 的初步 Schema（含 `perceptionMods` 字段用于投影计算参数）。

也指出 AI 生成的场景同样需要包装为 PF——引擎根据角色位置/GOAP 状态自动构建 PF，**预置和 AI 场景统一进入同一个 PF 管道**。

## 对话 11：投影计算的插件化

**KawOwl → Claude**
> 投影计算参数要仔细设计，如果过于复杂，可能会加重编剧的负担。我目前的想法是投影计算参数设计成可拓展的，里面是 Record<plugin, unknown>，每个 plugin 自己去设定如何解析它的参数。然后我们默认内置一个对人类友好的插件，它的计算逻辑是，当没有额外做标注时，每个人的内容投影和「我认为谁也知道这件事」同时 eavesdroppers 不被其他人所知。只要这个默认设置能覆盖 90% 的场景，编剧的工作压力将会被大大减轻。

**Claude → KawOwl**

确认了插件化架构：

- `perceptionMods: Record<string, unknown>`——键为插件名，值为该插件自定义参数
- **builtin 插件**（默认，零标注）：ratified/overhearers 拿完整内容 + 完整 knownPeers（不含 eavesdroppers），eavesdroppers 拿完整内容 + 完整 knownPeers（不含其他 eavesdroppers）
- 特殊插件按需加载（distance-degradation、content-override 等）

**编剧的工作量：只需列出谁在哪个角色位置上。90% 的场景零额外标注。**
